# Upgrade Guide: v0.1.x to v0.2.0

This guide covers breaking changes when upgrading to v0.2.0 of the MongoDB Atlas GCP module. The module remains in public preview; treat upgrades as you would any pre-v1 module.

- [backup_export: create_bucket renamed to create_gcs_bucket (Breaking Change)](#backup_export-create_bucket-renamed-to-create_gcs_bucket-breaking-change)
- [GCS Lifecycle Rules Added](#gcs-lifecycle-rules-added)
- [Backup Export IAM Role](#backup-export-iam-role)

## backup_export: create_bucket renamed to create_gcs_bucket (Breaking Change)

v0.2.0 renames the module-managed bucket object on `backup_export` from `create_bucket` to `create_gcs_bucket` for cross-CSP parity (AWS `create_s3_bucket`, Azure `create_storage_account`).

### Who is affected?

Users who set `backup_export.create_bucket` (module-managed or nested fields such as `enabled`, `location`, `name`).

### Required action

Rename the nested object. Nested field names are unchanged except for the parent key.

**Before (v0.1.x):**

```terraform
module "atlas_gcp" {
  source     = "terraform-mongodbatlas-modules/atlas-gcp/mongodbatlas"
  version    = "~> 0.1"
  project_id = var.project_id

  backup_export = {
    enabled = true
    create_bucket = {
      enabled  = true
      location = "us-east4"
    }
  }
}
```

**After (v0.2.0):**

```terraform
module "atlas_gcp" {
  source     = "terraform-mongodbatlas-modules/atlas-gcp/mongodbatlas"
  version    = "~> 0.2"
  project_id = var.project_id

  backup_export = {
    enabled = true
    create_gcs_bucket = {
      enabled  = true
      location = "us-east4"
    }
  }
}
```

### State migration

No Terraform state migration is required. Submodule resource addresses are unchanged; only the root variable path changes.

After updating your configuration:

1. Run `terraform plan`. The plan should show no infrastructure changes from the rename alone.
2. Run `terraform apply` when ready.

## GCS Lifecycle Rules Added

`expiration_days` is a new optional field on `backup_export.create_gcs_bucket`. When set to a value greater than zero, the module adds a `google_storage_bucket.lifecycle_rule` that deletes objects after that many days.

### Who is affected?

Users with module-managed backup export buckets. The plan shows a new lifecycle rule only when you set `expiration_days`.

### Required action

No action required if you want retention. Set `expiration_days` explicitly (typical value `365`) when you want a Delete rule:

```terraform
backup_export = {
  enabled = true
  create_gcs_bucket = {
    enabled         = true
    location        = "us-east4"
    expiration_days = 365
  }
}
```

### Zero-diff option

Omit `expiration_days` to keep v0.1.x behavior (no lifecycle rule on module-managed backup buckets). The field defaults to unset (`null`); the module does not add a lifecycle rule.

## Backup Export IAM Role

The module grants `roles/storage.objectUser` on the backup export GCS bucket (was `roles/storage.objectCreator`). Manual testing showed `objectCreator` is insufficient for Atlas backup snapshot export.

Log integration is unchanged and continues to use `roles/storage.objectCreator`.

### Who is affected?

- Users with module-managed backup export buckets (plan shows an in-place IAM role change on `google_storage_bucket_iam_member`).
- Users with `skip_iam_bindings = true` who pre-grant GCS roles externally.

### Required action

Allow the plan to update the module-managed IAM binding, or pre-grant `roles/storage.objectUser` on backup export BYO buckets before apply.

When `skip_iam_bindings = true`, pre-grant per feature:

- Backup export GCS buckets: `roles/storage.objectUser`
- Log integration GCS buckets: `roles/storage.objectCreator`

Do not use a single GCS role for both features.

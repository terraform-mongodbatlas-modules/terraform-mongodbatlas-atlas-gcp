## Configurable timeouts

v0.2.0 adds a nullable `timeouts` variable with 30m defaults on module-managed Atlas and GCP resources where the provider exposes timeout configuration.

### Who is affected?

All users upgrading from v0.1.x. The first plan may show in-place timeout block additions on supported resources. These updates are not destructive.

### Required action

No action required if the 30m defaults are acceptable. Allow the plan to add timeout blocks, then apply.

### Zero-diff option

Set `timeouts = null` to skip all module-managed timeout blocks and keep provider defaults:

```terraform
module "atlas_gcp" {
  source     = "terraform-mongodbatlas-modules/atlas-gcp/mongodbatlas"
  version    = "~> 0.2"
  project_id = var.project_id
  timeouts   = null
}
```

Remove or replace `timeouts = null` later if you want module 30m defaults.

## PrivateLink region key normalization (Breaking Change)

v0.2.0 normalizes `for_each` keys on module-managed multi-region PrivateLink endpoints to lowercase GCP region format (`us-east4`), even when input uses Atlas format (`US_EAST_4`).

### Who is affected?

Deployments where `privatelink_endpoints` used Atlas-format regions as Terraform state/plan keys.

Deployments that already used GCP-format regions (`us-east4`) skip migration.

### Required action

Add root-module `moved` blocks or run `terraform state mv` for each affected resource address. Update references to output map keys (for example `module.atlas_gcp.privatelink["US_EAST_4"]` → `["us-east4"]`).

**Example `moved` block** (repeat per endpoint key and nested resource address):

```terraform
moved {
  from = module.atlas_gcp.mongodbatlas_privatelink_endpoint.this["US_EAST_4"]
  to   = module.atlas_gcp.mongodbatlas_privatelink_endpoint.this["us-east4"]
}

moved {
  from = module.atlas_gcp.module.privatelink["US_EAST_4"]
  to   = module.atlas_gcp.module.privatelink["us-east4"]
}
```

**Example addresses:**

- `module.atlas_gcp.mongodbatlas_privatelink_endpoint.this["US_EAST_4"]` → `["us-east4"]`
- `module.atlas_gcp.module.privatelink["US_EAST_4"]` → `["us-east4"]` (moves nested GCP resources under the submodule)

**Not affected:** `privatelink_endpoints_single_region` (index keys), BYO Endpoint user-defined map keys.

The module does not ship bulk `moved` blocks.

## PrivateLink variable rename (Breaking Change)

v0.2.0 renames BYO Endpoint variables for cross-CSP parity with AWS and Azure v1 modules.

| v0.1.x | v0.2.0 |
| :---- | :---- |
| `privatelink_byoe_regions` (`map(string)`) | `privatelink_byo_endpoint` (`map(object({ region = string }))`) |
| `privatelink_byoe` | `privatelink_byo_service` |

### Who is affected?

Users who set `privatelink_byoe_regions` or `privatelink_byoe`.

### Required action

Rename variables and wrap each BYO Endpoint Region in an object. User-defined map keys are unchanged. No `moved` blocks are required.

**Before (v0.1.x):**

```terraform
privatelink_byoe_regions = { ep1 = "us-east4" }
privatelink_byoe = {
  ep1 = {
    ip_address           = google_compute_address.psc.address
    forwarding_rule_name = google_compute_forwarding_rule.psc.name
  }
}
```

**After (v0.2.0):**

```terraform
privatelink_byo_endpoint = { ep1 = { region = "us-east4" } }
privatelink_byo_service = {
  ep1 = {
    ip_address           = google_compute_address.psc.address
    forwarding_rule_name = google_compute_forwarding_rule.psc.name
  }
}
```

Region overlap with `privatelink_endpoints` is enforced after normalization via a plan-time precondition (not variable validation).

## Private endpoint regional mode (Breaking Change)

v0.2.0 no longer creates `mongodbatlas_private_endpoint_regional_mode` automatically when PrivateLink spans multiple Atlas service regions. A new `privatelink_regional_mode` input (`"auto"` or `"disabled"`, default `"disabled"`) opts in to the previous behavior.

### Who is affected?

Deployments that relied on automatic regional mode when `privatelink_endpoints`, `privatelink_endpoints_single_region`, or BYO Endpoint regions covered more than one distinct Atlas service region.

Deployments with a single PrivateLink region are unaffected.

### Required action

Set `privatelink_regional_mode = "auto"` when you need regional private connection strings (for example geo-sharded clusters where apps consume per-region URIs):

```terraform
module "atlas_gcp" {
  source                    = "terraform-mongodbatlas-modules/atlas-gcp/mongodbatlas"
  version                   = "~> 0.2"
  project_id                = var.project_id
  privatelink_regional_mode = "auto"
  privatelink_endpoints = [
    { region = "us-east4", subnetwork = google_compute_subnetwork.east.self_link },
    { region = "us-west4", subnetwork = google_compute_subnetwork.west.self_link },
  ]
}
```

Omit the variable or leave the default `"disabled"` when regional mode is not required.

Clusters that depend on regional mode should declare `depends_on = [module.atlas_gcp]` (or an explicit dependency on `mongodbatlas_private_endpoint_regional_mode`) so connection strings update before cluster resources apply.

## PSC consumer global access (optional)

Non-breaking. Omit `all_region_mode` on module-managed endpoint objects for same-region-only access (v0 behavior).

### Who needs this?

Deployments where VMs in other regions of the same VPC must reach a module-managed PSC endpoint IP in the Atlas service region.

### Module-managed endpoints

Set on the relevant `privatelink_endpoints` or `privatelink_endpoints_single_region` element:

```terraform
privatelink_endpoints = [
  {
    region                  = "us-east4"
    subnetwork              = google_compute_subnetwork.atlas.self_link
    all_region_mode = true
  }
]
```

The module sets `allow_psc_global_access` on the forwarding rule when `all_region_mode = true`. Do not use `allow_global_access` (internal load balancer only; invalid on PSC rules).

### BYO Endpoint

The module does not create the forwarding rule. Set `allow_psc_global_access` on your `google_compute_forwarding_rule` (or `gcloud compute forwarding-rules update ... --allow-psc-global-access`) before Phase 2 registration.

See [GCP global access](https://docs.cloud.google.com/vpc/docs/about-accessing-vpc-hosted-services-endpoints#global-access).

## backup_export: create_bucket renamed to create_gcs_bucket (Breaking Change)

v0.2.0 renames the module-managed bucket object on `backup_export` from `create_bucket` to `create_gcs_bucket` for cross-CSP parity (AWS `create_s3_bucket`, Azure `create_storage_account`).

### Who is affected?

Users who set `backup_export.create_bucket` (module-managed or nested fields such as `enabled`, `location`, `name`).

### Required action

Rename the nested object. Nested field names are unchanged except for the parent key.

**Before (v0.1.x):**

```terraform
module "atlas_gcp" {
  source     = "terraform-mongodbatlas-modules/atlas-gcp/mongodbatlas"
  version    = "~> 0.1"
  project_id = var.project_id

  backup_export = {
    enabled = true
    create_bucket = {
      enabled  = true
      location = "us-east4"
    }
  }
}
```

**After (v0.2.0):**

```terraform
module "atlas_gcp" {
  source     = "terraform-mongodbatlas-modules/atlas-gcp/mongodbatlas"
  version    = "~> 0.2"
  project_id = var.project_id

  backup_export = {
    enabled = true
    create_gcs_bucket = {
      enabled  = true
      location = "us-east4"
    }
  }
}
```

### State migration

No Terraform state migration is required. Submodule resource addresses are unchanged. Only the root variable path changes.

After updating your configuration:

1. Run `terraform plan`. The plan should show no infrastructure changes from the rename alone.
2. Run `terraform apply` when ready.

## GCS Lifecycle Rules Added

`expiration_days` is a new field on `backup_export.create_gcs_bucket` (default: **365** days). The module adds a `google_storage_bucket.lifecycle_rule` that deletes objects after that many days when the value is greater than zero.

### Who is affected?

Users with module-managed backup export buckets. The plan shows a new lifecycle rule on `google_storage_bucket`.

### Required action

No action required if the lifecycle rule is acceptable. To opt out, set `expiration_days = 0`:

```terraform
backup_export = {
  enabled = true
  create_gcs_bucket = {
    enabled         = true
    location        = "us-east4"
    expiration_days = 0
  }
}
```

### Zero-diff option

Set `expiration_days = 0` before upgrading. No lifecycle rule appears in the plan.

## GCS Versioning Default Changed

`versioning_enabled` on module-managed buckets defaults to `false` for `backup_export` and `log_integration` (was `true` in v0.1.x).

### Who is affected?

Users with `create_gcs_bucket.enabled = true` who omit `versioning_enabled`.

### Required action

Allow the plan to update `google_storage_bucket.atlas[0].versioning.enabled` from `true` to `false`, or set `versioning_enabled = true` explicitly to preserve the previous behavior:

```terraform
backup_export = {
  enabled = true
  create_gcs_bucket = {
    enabled            = true
    location           = "us-east4"
    versioning_enabled = true
  }
}
```

The same field applies under `log_integration.create_gcs_bucket`.

### Why the change?

Atlas writes unique object keys for backup exports and log delivery, so overwrite protection from versioning is unnecessary. This aligns with the AWS module default and avoids extra storage when lifecycle rules do not delete noncurrent versions.

## Backup Export IAM Role

The module grants `roles/storage.objectUser` on the backup export GCS bucket (was `roles/storage.objectCreator`). Manual testing showed `objectCreator` is insufficient for Atlas backup snapshot export.

Log integration is unchanged and continues to use `roles/storage.objectCreator`.

### Who is affected?

Users with backup export enabled (module-managed or user-provided `bucket_name`). The plan shows `google_storage_bucket_iam_member` **replaced** (`-/+`): changing `role` forces a new binding. Terraform destroys the old member and creates a new one.

Example:

```
# module.atlas_gcp.module.backup_export[0].google_storage_bucket_iam_member.atlas must be replaced
-/+ resource "google_storage_bucket_iam_member" "atlas" {
      ~ role = "roles/storage.objectCreator" -> "roles/storage.objectUser" # forces replacement
    }
```

### Required action

Allow the plan to replace the IAM binding with `roles/storage.objectUser`, then run `terraform apply`.

If you grant GCS IAM to the Atlas service account outside this module, use `roles/storage.objectUser` on backup export buckets. Log integration buckets still require `roles/storage.objectCreator`.
